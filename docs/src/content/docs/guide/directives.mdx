## Overview

AML directives are tags that instruct the runtime *how* to execute content, complementing `<skill>` tags which declare *what* to execute. There are three directive types:

| Directive | Purpose | Tag |
|---|---|---|
| Tool constraint | Restrict available tools within a scope | `<tool>` |
| Session isolation | Execute content in a separate session | `<session>` |
| Agent delegation | Delegate execution to a subagent | `<agent>` |

## `<aml>` — Document Root (Optional)

Wrap an entire AML document in an `<aml>` tag to declare the AML version:

```xml
<aml version="0.1">
  <skill name="analyser">analyse the codebase</skill>
</aml>
```

The `<aml>` wrapper is **optional**. When omitted, the parser operates in fragment
mode — extracting tags from arbitrary text. When present:

- `version` attribute is **required**
- Only whitespace and XML comments are allowed outside the wrapper
- `<aml>` cannot appear nested inside other tags
- Only one `<aml>` wrapper is allowed per document

### Attributes

| Attribute | Required | Description |
|---|---|---|
| `version` | Yes | AML specification version (e.g. `"0.1"`) |

## `<tool>` — Tool Constraints

Constrain which tools are available to skills within the scope:

```xml
<tool name="bash">
  <skill interface="run-tests">run the test suite</skill>
</tool>
```

### Attributes

| Attribute | Required | Description |
|---|---|---|
| `name` | No* | Single tool name — equivalent to `allow="<name>"` and narrows the constraint scope to that tool only |
| `allow` | No* | Comma-separated whitelist of allowed tools |
| `deny` | No* | Comma-separated blacklist of denied tools |

\* At least one of `name`, `allow`, or `deny` must be present. `allow` and `deny` are mutually exclusive.

### Examples

```xml
<!-- Allow only specific tools -->
<tool allow="bash,grep,view">
  <skill interface="search">find files matching pattern</skill>
</tool>

<!-- Deny specific tools -->
<tool deny="exec,net:write">
  <skill interface="analyse">review this code safely</skill>
</tool>
```

## `<session>` — Session Isolation

Execute content in a separate session with independent state:

```xml
<session name="backend-deploy">
  <skill interface="deploy">deploy the backend service</skill>
</session>
```

### Attributes

| Attribute | Required | Default | Description |
|---|---|---|---|
| `name` | No | — | Session identifier |
| `isolated` | No | `"true"` | Whether session state is isolated from the parent |
| `on-failure` | No | `"halt"` | `"halt"`, `"skip"`, or `"partial"` — controls child failure propagation |

### Examples

```xml
<!-- Isolated session (default) -->
<session name="analysis">
  <skill interface="heavy-computation">process large dataset</skill>
</session>

<!-- Non-isolated session (shared state) -->
<session name="shared-work" isolated="false">
  <skill interface="update-state">modify shared context</skill>
</session>
```

## `<agent>` — Subagent Delegation

Delegate execution to a specialised subagent:

```xml
<agent name="security-reviewer" model="gpt-4">
  <skill interface="security-audit">
    Review this authentication module.
  </skill>
</agent>
```

### Attributes

| Attribute | Required | Default | Description |
|---|---|---|---|
| `name` | Yes | — | Agent identifier (resolved against host registry) |
| `model` | No | — | Model override (request, not guarantee) |
| `mode` | No | `"sync"` | `"sync"` (blocking) or `"background"` (async) |
| `on-failure` | No | `"halt"` | `"halt"`, `"skip"`, or `"partial"` — controls child failure propagation |

### Examples

```xml
<!-- Synchronous agent (blocks until done) -->
<agent name="code-reviewer">
  Review this pull request for correctness.
</agent>

<!-- Background agent (runs asynchronously) -->
<agent name="indexer" mode="background">
  Index all files in the repository.
</agent>

<!-- Agent with model preference -->
<agent name="architect" model="claude-opus">
  Design the microservice architecture for this system.
</agent>
```

## Nesting

Directives nest freely with each other and with `<skill>` tags:

```xml
<session name="secure-review">
  <agent name="reviewer">
    <tool allow="grep,view">
      <skill interface="code-review" language="python">
        def handle_payment(card_number, amount):
            return process(card_number, amount)
      </skill>
    </tool>
  </agent>
</session>
```

This means: create a separate session, delegate to the reviewer agent, restrict to grep and view tools, then invoke the code review skill.

## Self-Closing Directives

Directives support self-closing syntax for empty scopes:

```xml
<tool name="bash" />
<agent name="monitor" mode="background" />
```

## Security

- `<tool>` can only **narrow** tool access — never expand beyond host policy.
- `<agent>` names are resolved against the host's agent registry.
- `<session>` creation is subject to host authorisation.
- `model` on `<agent>` is a request — the host may override or deny it.
- Directives inside definition bodies (`define="interface"` or `define="implementation"`) are invalid.

## Failure Handling

The `on-failure` attribute on `<session>` and `<agent>` controls how child
failures propagate:

```xml
<!-- Skip failures silently -->
<session name="best-effort" on-failure="skip">
  <skill interface="optional-check">may fail safely</skill>
</session>

<!-- Capture failures inline -->
<agent name="reviewer" on-failure="partial">
  <skill interface="code-review">review this code</skill>
</agent>
```

| Value | Behaviour |
|---|---|
| `halt` (default) | Propagate failure immediately — abort the directive |
| `skip` | Suppress failures — produce output only from successful children |
| `partial` | Capture failures inline as `[DIRECTIVE FAILED: ...]` markers |

`<tool>` does not support `on-failure` — it constrains scope, not execution.

## Tool Constraint Composition

Nested `<tool>` directives compose monotonically. Inner scopes can only
further restrict, never widen, the tool set:

```xml
<!-- Outer allows 4 tools; inner narrows to 2 -->
<tool allow="grep,view,bash,glob">
  <tool allow="grep,view">
    <!-- Only grep and view available here -->
    <skill interface="search">query</skill>
  </tool>
</tool>

<!-- Deny at any level is permanent -->
<tool deny="bash">
  <tool allow="bash,grep">
    <!-- bash is denied by ancestor — only grep is effective -->
    <skill interface="search">query</skill>
  </tool>
</tool>
```

| Pattern | Rule |
|---|---|
| Allow ∩ Allow | Intersection — tool must be allowed at every level |
| Deny ∪ Deny | Union — denied at any level means denied everywhere below |
| Deny beats Allow | An inner allow cannot reinstate a tool denied by an ancestor |

> **Security note:** `bash` is effectively a superuser tool — an agent with
> bash access can bypass file-level tool constraints using shell commands
> (e.g. `sed -i` instead of `edit`). For read-only review contexts,
> prefer `allow="grep,view"` without `bash`.
