## Concepts

- **Interface**: An abstract capability (e.g. "unit-testing")
- **Implementation**: A concrete realisation (e.g. "pytest-runner")

## Defining Interfaces

```xml
<skill define="interface" name="unit-testing">
  Execute automated tests and report pass/fail results with coverage.
</skill>
```

### Typed Interface Declarations

Interfaces can optionally declare typed parameters, return values, and file I/O patterns — making the contract self-documenting and machine-verifiable.

```xml
<skill define="interface" name="brain-query">
  <param name="question" type="string" required="true">The question to answer</param>
  <param name="format" type="enum" values="markdown|comparison-table|slide-deck|chart" default="markdown">Output format</param>
  <param name="caller_session_id" type="string" required="false">Set by brain-responder</param>
  <returns name="answer" type="string">Citation-backed answer</returns>
  <returns name="quality" type="enum" values="answered|partial|unanswered" />
  <reads>wiki/index.md, wiki/**/*.md</reads>
  <writes>raw/brain-questions.md</writes>
</skill>
```

#### Parameter types

| Type | Description |
|------|-------------|
| `string` | Free-form text (default when `type` is omitted) |
| `enum` | One of the pipe-separated `values` (e.g. `values="a\|b\|c"`) |
| `number` | Numeric value |
| `boolean` | `"true"` or `"false"` |
| `path` | File or directory path |
| `list` | List of values |

#### Attributes

- **`required`** — `"true"` or `"false"` (default: `false`)
- **`default`** — Default value when the parameter is not provided
- **`values`** — Required for `enum` type; pipe-separated allowed values

#### File I/O declarations

`<reads>` and `<writes>` declare which files the skill reads from or writes to, using comma-separated glob patterns. This replaces the `code_in`/`code_out` naming with honest, descriptive semantics.

> **Backward compatibility:** Interfaces with text-only bodies (no typed declarations) remain fully valid.

### Skill References and Tool Constraints

Interfaces can reference other skills (e.g. for DDE enforcement) and declare tool constraints as part of the contract. `<skill ref>` supports both self-closing and wrapping forms — see the [Best Practices](/guide/best-practices/) guide for the wrapping pattern.

```xml
<skill define="interface" name="brain-query">
  <param name="question" type="string" required="true">The question to answer</param>
  <returns name="answer" type="string">Citation-backed answer</returns>
  <reads>wiki/index.md, wiki/**/*.md</reads>
  <writes>raw/brain-questions.md</writes>
  <skill ref="dde" role="enforcement" />
  <tool allow="rename_session,send_session_message,view,edit,bash" />
</skill>
```

- **`<skill ref="..." role="...">`** — References another skill. `role` is optional and describes the relationship (e.g. `"enforcement"` for DDE).
- **`<tool allow="...">`** or **`<tool deny="...">`** — Declares which tools the skill is permitted (or forbidden) to use. Self-closing only; `allow` and `deny` are mutually exclusive.

## Defining Implementations

```xml
<skill define="implementation" name="pytest-runner" implements="unit-testing" language="python" framework="pytest">
  Run tests using pytest with coverage via pytest-cov.
</skill>

<skill define="implementation" name="jest-runner" implements="unit-testing" language="javascript" framework="jest">
  Run tests using Jest with coverage reporting.
</skill>
```

### DDE Node Declarations

Implementation bodies can include structured `<node>` declarations that describe each step in a DDE (Diagram-Driven Execution) flowchart:

```xml
<skill define="implementation" name="brain-query-impl" implements="brain-query">
  ```mermaid
  flowchart LR
    A[Rename] --> B[ReadIndex] --> C[Synthesise] --> D[Reply]
  ```
  <node name="Rename" type="tool">
    <tool use="rename_session" />
    Rename this session with pattern `query-<brief-description>`
  </node>
  <node name="ReadIndex" type="tool">
    <tool use="view" />
    Read wiki/index.md to find candidate pages
  </node>
  <node name="Synthesise" type="prompt">
    Drill into candidate pages, synthesise citation-backed answer
  </node>
  <node name="Reply" type="tool">
    <tool use="send_session_message" />
    Send the answer back to the caller
  </node>
</skill>
```

#### Node attributes

- **`name`** — Must match the mermaid flowchart label exactly
- **`type`** — `"tool"` for deterministic tool invocations, `"prompt"` for LLM reasoning steps
- **`<tool use="...">`** — Optional child element declaring which specific tool a `tool`-type node invokes

> **Note:** Node names must be unique within an implementation. Tool-type nodes should declare `<tool use="...">`, and prompt-type nodes should not.

## Invoking by Interface

```xml
<skill interface="unit-testing" language="python">
  <param name="target">src/auth.py</param>
  Run tests for this module.
</skill>
```

The runtime resolves `unit-testing` + `language="python"` → `pytest-runner`.

## Resolution Priority

1. `impl` specified → use that exact implementation
2. `interface` specified → filter by hints, pick highest priority
3. `name` specified → direct lookup

## Handling Ambiguity

If multiple implementations match with equal priority, resolution **fails explicitly** with an `Ambiguous` error. This is intentional — AML never silently picks a random implementation.
